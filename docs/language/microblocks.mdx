# MicroBlocks


<div style={{ textAlign: 'center'}}>
![MicroBlocks](pathname:///img/microblocks.webp)
</div>

MicroBlocks is a blocks language (like Scratch) that makes it easy for beginners (9 years to adults) to program DUELink boards. A huge benefit of MicroBlocks is that it is live and
interactive: users can click commands to see what they do and they can test their scripts as they work.

<div style={{ textAlign: 'center'}}>
![MicroBlocks](pathname:///img/microblocks-sample.webp)
</div>

Yet in spite of its apparent simplicity, MicroBlocks is a fast, powerful programming language with the ability to interface with peripherals via GPIO, I2C, SPI, and serial
and nearly 200 extension libraries. In fact, some hardware designers prefer MicroBlocks for rapid prototyping and testing because of its fast, interactive development cycle.

There are two ways to use MicroBlocks with DUELink:
1. Run MicroBlocks directly on DUELink modules/boards **Standalone**.
2. Use MicroBlocks to control **Tethered** Modules.

:::tip
Most product pages include Tethered and Standalone sample projects.
:::

An example of Standalone is when loading and running MicroBlocks directly on boards like [DueSTEM](../products/mcduestem-b) or [Ghizzy](../products/spghizzy-d). 

<div style={{ textAlign: 'center'}}>
  <iframe  width="300" height="500" text-align="center"
  src="https://www.youtube.com/embed/SRl_iSj7iYo" 
  frameborder="10px"
  allowfullscreen="true"></iframe>
</div>
<br/>

A tethered example can be [micro:bit](../hw/microbit) that is controlling [LED RGB3](../products/ldledrgb3-a). The [LinkBit](../products/adlinkbit-a) adaptor is here to give micro:bit the needed [Downlink](../interface/downlink) interface.

<div style={{ textAlign: 'center'}}>
![Linkbit](pathname:///img/microbit-linkbit.webp)

It is also possible to combine both options! In this case, MicroBlocks will run Standalone on a DUELink board, that is also Tethered to one, or more, DUELink modules. This can for example be [CincoBit](../products/mccincobit-e) running MicroBlocks firmware and user blocks Standalone that is controlling [LED RGB3](../products/ldledrgb3-a) module.

## MicroBlocks Standalone

While you can load MicroBlocks to make any DUELink module run Standalone, the ones recommended on the [educators](../educators) page make it especially easy to get started.

### Installing the Firmware

Educational-tailored boards automatically ship with MicroBlocks preloaded. You can skip this section in most cases. However, here are the steps needed to load MicroBlocks firmware, a process that takes less than a minute.

Start by connecting your DUELink board to a computer using a USB cable. Make sure it is not a charge-only cable!

:::note
Linux may require an extra step to ensure that the user has access to USB serial devices.
See the Linux tab in the Computer Setup section of the [MicroBlocks Get Started](https://microblocks.fun/get-started) page.
:::

The board must be placed in a special DFU (Device Firmware Update). This mode can be entered using the `Erase All` option found in the `Firmware` top-menu on [console.duelink.com](https://console.duelink.com/). The [Loader](../loader) page has full details if needed.

<div style={{ textAlign: 'center'}}>
![erase all](pathname:///img/firmware-eraseall.webp)
</div>

The board should now have no firmware installed and and is in DFU mode. If you are in using Windows and want to double check, you can check the `Device Manager`.

<div style={{ textAlign: 'center'}}>
![erase all](pathname:///img/device-manager-dfu.webp)
</div>

You also have the option to force DFU mode without erasing the firmware by holding `LDR` or `A` buttons during a module reset or power up. 

The process is similar for all educational-tailored microcomputer boards. For example, prepare CincoBit to accept firmware by holding down the `A` button while clicking the `RESET` on the back of the board:

<table>
<td width='50%'>
![CincoBit A Button](pathname:///img/cincobit-a-button.webp)
</td>
<td width='50%'>
![CincoBit Reset](pathname:///img/cincobit-reset.webp)
</td>
</table>

On DueSTEM, the button is labeled `LDR` (left-most red button). Hold it while clicking the `Reset` button:

<div style={{ textAlign: 'center'}}>
![duestem Reset](pathname:///img/duestem-reset.webp)
</div>

ClipIt is done similarly to DueSTEM:

<div style={{ textAlign: 'center'}}>
![clipit Reset](pathname:///img/clipit-reset.webp)
</div>

With your board connected in DFU mode (via `Erase All` or using buttons), open [console.duelink.com](https://console.duelink.com/) in a Chrome or Edge browser:

<div style={{ textAlign: 'center'}}>
![dueconsole](pathname:///img/duelink-console.webp)
</div>

Click on the **Firmware** tab:

<div style={{ textAlign: 'center'}}>
![firmware](pathname:///img/console-firmware.webp)
</div>

In the dialog that appears, click the `Connect` button. You should see a dialog box with an entry marked **DFU in FS Mode**:

<div style={{ textAlign: 'center'}}>
![console](pathname:///img/console-connect-dfu.webp)
</div>

If you do not see the dialog box, make sure you are using a Chrome or Edge browser. Other browsers cannot connect to the board.

If you do not see see **DFU in FS Mode**, repeat the board preparation process described above. If using Windows, you can check the `Device Manager` as your PC may needs the DFU drivers. Download and install the USB drivers from the [downloads](../downloads) page.

Select the **DFU in FS Mode** entry and click the `Connect` button.

In the next dialog, select **MicroBlocks (microblocks.fun)** in the Firmware menu. The version menu will default to the most recent version, which is what you want:

<div style={{ textAlign: 'center'}}>
![console](pathname:///img/console-connect-3.webp)
</div>

Click the `Load` button.
You'll see progress as the board is erased and then the firmware is written:

<div style={{ textAlign: 'center'}}>
![progress-erasing](pathname:///img/progress-erasing.webp)
</div>
<div style={{ textAlign: 'center'}}>
![progress-writing](pathname:///img/progress-writing.webp)
</div>

Close the `Console` browser tab. Congratulations! You've installed MicroBlocks, you're ready to start coding!

You don't need to install MicroBlocks firmware every time you use MicroBlocks, only if you want to update to a newer version or switch between MicroBlocks and DUELink firmware.

:::note
Although this section focused on installing MicroBlocks on the educational microcomputer boards, MicroBlocks can be installed on *any* DUELink board. See [MicroBlocks Everywhere!](#microblocks-everywhere).
:::

### Coding in MicroBlocks

MicroBlocks is a **live** coding environment so you can test as you code
and try out blocks to discover what they do.

Click on a block or script to run it:

<div style={{ textAlign: 'center'}}>
![Smile and Frown](pathname:///img/microblocks-instant.gif)
</div>

Drag blocks into the scripting pane and assemble them to create scripts.

Your code is stored in persistent flash memory and can run even when the board is not connected to the MicroBlocks editor.

Scripts starting with a **when started** block will run when the board is powered on.

<div style={{ textAlign: 'center'}}>
![Smile and Frown](pathname:///img/microblocks-whenstarted.webp)
</div>

MicroBlocks supports concurrency. Up to ten scripts can run at the same time.

### MicroBlocks Pin Numbers

On the CincoBit and PixoBit, MicroBlocks uses the same edge connector pin numbers as the [micro:bit](https://tech.microbit.org/hardware/edgeconnector/).
Pins 0, 1, and 2 are the three alligator clip connections with holes on the left side of the edge connector.

On the CincoBit:
- pin 9 is the analog light sensor
- pin 21 is the buzzer

On the PixoBit:
- pin 21 is the buzzer
- pin 22 is the display reset pin
- pin 23 is the analog light sensor

On all other DUELink boards, MicroBlocks uses standard DUELink pin numbers.
Pins 0-9 and 17 are capable of analog input.
The pin numbers are marked on the DueSTEM and most other microcomputer boards.
You can always find the pin numbers for a given board on its schematic.

## Tethering Modules

MicroBlocks can control Tethered DUELink modules. It can be a single module, or a [Daisylink](../engine/daisylink) of modules. When you do this, only the first module in the chain runs the MicroBlocks firmware. All other modules in the chain run the standard DUELink firmware and respond to commands from MicroBlocks.

<div style={{ textAlign: 'center'}}>
![PixoBit Daisylinked](pathname:///img/pixobit-daisylinked.webp)
</div>

### Tethering Options

If you are running MicroBlocks on a DUELink board/module, connect the Downlink (**D**) connector on that board.

<div style={{ textAlign: 'center'}}>
![Cincobit downlink port](pathname:///img/cincobit-downlink.webp)
</div>

to the Uplink (**U**) connector of the next DUELink module.

<div style={{ textAlign: 'center'}}>
![RGB3 uplink port](pathname:///img/rgb3-uplink.webp)
</div>

<div style={{ textAlign: 'center'}}>
![animated GIF of flashing LEDs](pathname:///img/microblocks-daisylinked.gif)
</div>

If you are running MicroBlocks on a non-DUELink board, connect the Daisylink module chain to your board's I2C or UART pins. If you have a [micro:bit](../hw/microbit), you can use a [LinkBit](../products/adlinkbit-a) to tether DUELink modules.

<div style={{ textAlign: 'center'}}>
![Linkbit](pathname:///img/microbit-linkbit.webp)
</div>

When using multiple DUELink modules, modules are daisy-chained together by connecting the Downlink (**D**) connector of one module to the Uplink (**U**) connector of the next (see [Daisylink](../engine/daisylink)).

<div style={{ textAlign: 'center'}}>
![Uplink/Downlink Connections](pathname:///img/mb-uplink-downlink.webp)
</div>

:::tip
Always connect a Downlink (**D**) connector to an Uplink (**U**) connector.
Nothing will work if a **D** is connected to a **D** or a **U** to a **U** anywhere in the chain.
:::

:::tip
Once all connections have been made, power cycle the entire chain to reset all modules before attempting to use any module.
:::

### Daisylink Library

We maintain a set of libraries to simplify the use of MicroBlocks with DUELink. `DUELink Daisylink` is of these libraries and it is used to access and control Tethered modules, which can be one module or a Daisylink of modules. In either case, we need this `DUELink Daisylink` library.

In the MicroBlocks editor, click the **Add Library** button:

<div style={{ textAlign: 'center'}}>
![add-library](pathname:///img/add-library.webp)
</div>

then from `Libraries` select `Kits and Boards` and then `DUELink Daisylink` library:

<div style={{ textAlign: 'center'}}>
![computer-shortcut](pathname:///img/microblocks-lib-daisylink.webp)
</div>

In this example setup, `CincoBit` is the board running Standalone and MicroBlocks is loaded and is running on it. The tethered `LED RGB3` is is running the official DUELink firmware and is controlled by commands coming from `CinoBit`.

From the `DUELink Daisylink` library palette click on the `DUELink connected?` block to verify that your `LED RGB3` is connected and detected by `CincoBit`:

<div style={{ textAlign: 'center'}}>
![connected-test](pathname:///img/connected-test.webp)
</div>

In this Daisylink example, there is only one device, which is `LED RGB3`, and it is address 1. If you have another module connected downlink to `LED RGB3` it will become device address 2. You can use `select device` block to select the desired device, and then use the `DUELink connected?` block again to verify that the selected device is functional.

:::tip
Device address 1 is automatically selected on power up. If you only need to use device address 1, there will be no need to use the `select device` block.
:::

### Module Drivers

Modules that are meant to be commanded in a Dasiylink, like `LED RGB3` ship preloaded with DUELink official firmware and[Driver Script](../engine/drivers). The driver gives you a set of functions that you can access externally, without needing to understand how the module works nor needing to look at schematics. Those functions are documented under the individual product pages, under the `Driver` tab.

In this case, `LED RGB3` includes `SetLed(i, c)` function, which sets an LED at `i` index (0 to 2) to `c` a hexadecimal RRGGBB color (e.g. FF0000 is red).

The `DUELink Daisylink` library includes helper blocks to build these function. Here is an example of a helper block `set RGB3 led` that takes the 2 needed argument, the LED index and its color.

## MicroBlocks Everywhere!

We recommend starting with MicroBlocks on the boards recommended for [educators](../educators). From there, all other modules can be daisylinked as shown in the previous section. However, it is possible to run MicroBlocks on any DUELink module, allowing it to run Standalone! For example, you can install MicroBlocks on the [LED RGB3](../products/ldledrgb3-a) and control the LEDs directly from MicroBlocks.

Since [LED RGB3](../products/ldledrgb3-a) does not have a USB connector, you will need an adapter such as [USB Hook](../products/adusbhook-a).

<div style={{ textAlign: 'center'}}>
![USB Hook for LED RGB3](pathname:///img/usbhook-rgb3.webp)
</div>

Just like before, to install MicroBlocks firmware on a board, you need to set the board in DFU mode. You can do so using the `Erase All` feature explained earlier. The other option was to force DFU mode using the LDR button. These boards do not have an LDR button but you can accomplish the same results by inserting a small metal paper clip (or a wire) between 2 special small pads, while powering up the board. See the [Loader](/docs/loader) page for full details.

<div style={{ textAlign: 'center'}}>
![LDR TH pads](pathname:///img/module-loader-pads.webp)
</div>

Please note, once once you have MicroBlocks firmware installed on the `RGB3` module, you no longer have the DUELink firmware and module Driver Script. Controlling the module's hardware must be done 100% with MicroBlocks code. For modules with simple hardware such as LEDs, buttons, and analog sensors, the MicroBlocks code is easy to write. However, more complex modules such as displays and SD card readers are best used as Daisylinked modules so one can take advantage of the hardware support built into the DUELink standard firmware. For example, instead of loading MicroBlocks directly on the [SD Card](../products/stsdcard-b) module and worrying about file system and SD drivers, use MicroBlocks firmware on [CincoBit](../products/mccincobit-e), which is then connected to the [SD Card](../products/stsdcard-b) module that is running the DUELink standard firmware and Driver Script.

<div style={{ textAlign: 'center'}}>
![Cincobit and SD Card](pathname:///img/cincobit-sdcard.webp)
</div>

### Additional Libraries

When connected to one of the [educational](../educators) boards, MicroBlocks adds the relevant libraries automatically. For example, when a new project is created while connected to a `CincoBit` board, MicroBlocks adds the `LED Display` library.

Beyond what is already included in MicroBlocks editor, we also provide additional MicroBlocks libraries. Download the library you need from the [Downloads](../downloads) page and add it to your MicroBlocks project by using the **Add Library** button or by dropping the library file onto the MicroBlocks editor.

## Linux Support

DUELink modules loaded with MicroBlocks firmware fully work with Linux. However, in order for MicroBlocks to work, you need to have permission to access the serial port. Linux users are expected to know how to do so but here is an example sell script to get you started.

```
#!/bin/bash

echo "Adding user to dialout group for serial port access..."
sudo usermod -aG dialout $USER

# Check if user was added to dialout group
if id -nG "$USER" | grep -qw "dialout"; then
    echo "User $USER added to dialout group successfully."
else
    echo "Failed to add user to dialout group. Please check permissions."
    exit 1
fi

echo "Creating udev rule for STM32 DFU mode..."

# Add the udev rule for STM32 DFU mode (Vendor ID: 0483, Product ID: df11)
echo 'SUBSYSTEM=="usb", ATTR{idVendor}=="0483", ATTR{idProduct}=="df11", MODE="0666"' | sudo tee /etc/udev/rules.d/99-stm32-dfu.rules > /dev/null

# Reload udev rules to apply the changes
sudo udevadm control --reload-rules

echo "Permissions set successfully!"
echo "For changes to take effect:"
echo "- You need to log out and log back in for the group changes (serial port access) to take effect."
echo "- After logging in again, you should have access to both serial and DFU mode without requiring sudo."

echo "If you are unsure about how to log out, please simply restart your system."
```
Save the above script into a `.sh` file and give it executable permissions `(chmod +x)`. Now, you can run the `.sh` file and then reboot (or re-login), for the system to take effect.
