# Graphics
---

The graphics support is versatile and support drawing on displays and LEDs of many types.

## Configuration

Before any drawing operations, the systems needs ot know what device it will access and what pixel mapping it needs to process internally.

`GfxConfig (type, cfg1, cfg2, width, height, direct, multiplier)` Is needed to set the device `type` with `width` and `height`. The `cfg1` and `cfg2` are optional and depend on the device type.

`direct` mode is 1 when drawing to the display directly. No need for internal buffering and no need for `Show()`. This is useful to draw to 320x240 displays at full color. The system memory is limited and can't process 320x240 at 16BPP, which requires 150K of RAM! Instead, set direct to 1 and the graphics engine will draw directly to the display.

Another options for larger display is to use `multiplier`. In this case, the engine will multiply each pixel, 2 or 3 times, to cover the entire screen. For example, 160x128 display with x2 multiplier requires only 80x64 of memory.

When selecting to use graphics with `direct` set to 0 (use internal buffering), make sure the overall memory need is under 10KBytes. 320x240 with 3 x multiplier is 106x80. We typically set it to 100x80 pixels, requiring 8KBytes of RAM.

:::tip
Any buffered graphics are done using 8BPP (256 colors). Use direct mode if more colors are required. 
:::

Supported device types:

| Type | Description
| --- | ---
| 0 | None (default)
| 1 | I2C device, with `cgf1` being the salve address. `cfg2` is ignored.
| 2 | SPI device, with `cfg1` being the chip select and `cfg2` is data control pins.
| 3 | Neopixel WS2812 support. `cfg1` is the data pin.
| 4 | 5x5 LED matrix. The pins are fixed. Note that all argument are ignored with this type as width and height are set to 5, with no buffering and no multiplier.

:::tip
The graphics libraries does not include any device initialization. An application must initialize the device as necessary and then `GfxConfig()` before using any drawing functions.

## SPI TFT Display

```py
dim b1[2]
_c=0
Init()
while 1
    Text(Str(_c), 0xFF00FF, 30, 5)
    Show()
    _c=_c+1    
    wait(100)
wend

fn Init()
	dwrite(11,1) # reset pin
    # config SPI
    SpiConfig(...)
    # 80x84 with x2 (160x128) using buffered drawing
    Gfxconfig(2,3,4,80,64, 1, 2) 
	Wait(20)
fend
```

## I2C OLED Display

In this case, we are using 

```py
dim b1[2]
_c=0
Init()
while 1
    Text(Str(_c), 0xffFFFF, 30, 5) # you can also just set color to 1 since this is B&W
    Show()
    _c=_c+1    
    wait(100)
wend

fn Init()
	dwrite(11,1) # reset pin

    # config I2C bus with 400Kz
    i2ccfg(400)
    Gfxconfig(1,0x3C,0,128,64, 0, 1)
	Wait(20)
 
	SendCmd( 0xAE)
	SendCmd( 0x00)
	SendCmd( 0x10)
	SendCmd( 0x40)
	SendCmd( 0x81)
	SendCmd( 0xCF)
	SendCmd( 0xA1)
	SendCmd( 0xA6)
	SendCmd( 0xA8)
	SendCmd( 0x3F)
	SendCmd( 0xD3)
	SendCmd( 0x00)
	SendCmd( 0xD5)
	SendCmd( 0x80)
	SendCmd( 0xD9)
	SendCmd( 0xF1)
	SendCmd( 0xDA)
	SendCmd( 0x12)
	SendCmd( 0xDB)
	SendCmd( 0x40)
	SendCmd( 0x8D)
	SendCmd( 0x14)
	SendCmd( 0xAF)
	SendCmd( 0xC8)
	SendCmd( 0x20)
	SendCmd( 0x00)

fn SendCmd(c)
	b1[0] = 0
	b1[1] = c
    i2cmw(0x3c, b1,2, 2000)    
fend
```

## 5x5 LED Matrix

This is a perfect match for [CincoBit](../catalog/mainboard/cincobit) with its 5x5 LED matrix.

```py
Gfxconfig(4,0,0,0,0,0,0) # all arguments are ignored in 5x5 LED type

_x = 6
# We need tiny font as we only have 5 pixels!
while 1
    TextT("DUELink", 0xffFFFF, _x, 0) # you can also just set color to 1 since this is B&W
    Show()
    _x=_+1
    if _x < -20
        _x = 6
    end    
    wait(100)
wend

```
